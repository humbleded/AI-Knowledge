# 采样参数与成本：temperature / top-k / top-p 与 token 计费

## 一句话解释

模型每写一个 token 都在「**按概率抽签**」；采样参数（temperature / top-k / top-p）调这个抽签箱的**随机性**与**候选范围**；成本则按**输入 + 输出的 token 量**计费。

## 地基：写字 = 抽签

模型每生成一个 token，分两步：

1. 给一堆候选 token **打分**（原始分 = logits，公式里记作 `z_i`，`i` 是「第几个」的编号）。
2. 用 **Softmax** 把分数换成「**中签概率**」（加起来 = 100%，分高签多但**不包中**），再**按概率抽**一个。

- Softmax = 「把分数翻译成百分比」的工具。不用会算，记住：**分高 → 概率大，但不清零、和为 1**。
- 这就是「同一问题问两次、答案不一样」的根源——它在抽签，不是照背。
- 概率最高的 token **不一定**被选中，只是「签最多」。

## 三个旋钮

| 旋钮 | 管什么 | 大白话 |
|---|---|---|
| **temperature** | 改概率差距 | 抽签箱**死板还是放飞** |
| **top-k** | 砍候选名单（固定个数） | 只留**前 k 个**，其余扔 |
| **top-p** | 砍候选名单（动态） | 留到**累计够 p%**，其余扔 |

### temperature（最常用）

只做一件事：调候选之间「**概率差距悬殊度**」。

- **低温** = 差距拉大 = 分布**陡** = 稳定 / 可复现 / 死板重复。
- **高温** = 差距压平 = 分布**平** = 多样 / 创意 / 可能不连贯。

公式 `p_i = e^(z_i / T) / Σ e^(z_j / T)` 翻成人话：跟 Softmax 是**同一个东西**，只多了「每个分数先**除以 T**」。`e^(...)` 是个「**放大差距的放大器**」（分数差一点、签数差很多）：

- 除以**小 T** → 进放大器前先把分数撑大 → 差距被拉得更狠 → 陡 → 死板。
- 除以**大 T** → 分数被压小 → 差距抹平 → 平 → 放飞。
- `T=1` 啥也不改；`T=0` = 无限陡 = **永远只抽最高分** = 贪心 greedy = 每次一样。

三档示意（候选：好/热/冷/香蕉）：

| 候选 | 低温 T=0.2 | 默认 T=1.0 | 高温 T=1.5 |
|---|---|---|---|
| 好 | 99% | 70% | 50% |
| 热 | 1% | 20% | 27% |
| 冷 | ~0% | 9% | 18% |
| 香蕉 | 0% | 1% | 5% |

### top-k / top-p（砍候选名单）

- **top-k**：按概率从高到低，**只留前 k 个**，其余扔，再从中抽（固定个数）。
- **top-p（核采样）**：从高到低**累加概率**，到累计够 `p`（如 0.9）就停，留这一撮（**个数动态**——模型越确定名单越短，越犹豫名单越长）。
- **关键等价**：`k=1` ≡ `T→0` ≡ 贪心（最死板、每次一样）。
- 通常 **top-k / top-p 二选一**即可；实战主调 temperature。

## DeepSeek 推荐温度（数字不通用！）

默认 `temperature = 1.0`。官方按任务推荐：

| 任务 | 推荐温度 |
|---|---|
| 代码生成 / 数学解题 | **0.0** |
| 数据抽取 / 分析 | 1.0 |
| 通用对话 | 1.3 |
| 翻译 | 1.3 |
| 创意写作 / 诗歌 | **1.5** |

> 记忆：**要稳要准 → 往 0 拧；要花要野 → 往 1.5 拧。**

- ⚠️ **同一个数字在不同厂商模型上不是一回事**（DeepSeek 拿 1.3 当日常对话，Hello-Agents 书里 0.7 就算「高温」了）。**直觉通用**（低=稳、高=野），但**具体拨几要看你用的那个模型自己的推荐表**，别跨厂商硬套。
- DeepSeek **思考模式**下采样行为不同，做温度对比实验建议用**非思考模式**（删掉 `reasoning_effort` / `thinking`），温度效果才干净、也不白烧 token。

## token 成本构成

钱 = **输入 token + 输出 token**（按量 × 单价，单位：元 / 百万 token）：

- **输入** = `system` + 历史 history + 本轮问题；**输出** = 模型生成的回答。
- **缓存命中 / 未命中**：输入开头有一段和之前请求重复（固定 system、重复历史）→「命中」→ 极便宜。

| 项 | deepseek-v4-flash | deepseek-v4-pro |
|---|---|---|
| 输入（缓存命中） | 0.02 | 0.025 |
| 输入（缓存未命中） | 1 | 3 |
| 输出 | 2 | **6** ← 最贵 |

（旧名 `deepseek-chat` / `deepseek-reasoner` 2026-07-24 弃用，对应 v4-flash 的非思考 / 思考模式。）

**省钱三招（成本意识）：**
1. **输出最贵** → 别让模型啰嗦，能短就短。
2. **固定 system / 重复前缀走缓存**（0.025，近乎白送）→ 别乱改。
3. **history 越堆越长 → 输入越大 → 越贵** → `trim_history` 不只是为上下文窗口，**也是省钱**。

**看一次花了多少 token：** 非流式返回里 `response.usage` → `prompt_tokens`（输入）/ `completion_tokens`（输出）/ `total_tokens`。
**温度不改单价**，但高温可能让回答更长 → 输出 token 更多 → **间接更贵**。

## 常见坑 / 错误理解 → 正确理解

- ❌「`k=1` / `T→0` 最死板」≈ 高温 → ✅ 恰恰**相反**，它等于**低温到底 / 贪心**；判断方向先问自己「这是更**死板**还是更**放飞**」。
- ❌ 跨厂商套同一个 temperature 数字 → ✅ 看**你用的模型自己**的推荐表。
- ❌ 以为概率最高的 token 一定被选中 → ✅ 只是「签最多」，会抽中别的。
- ❌ 以为 temperature 影响价格 → ✅ 不改单价，只可能因回答变长而**间接**多花输出 token。
- ❌ 开着思考模式做温度实验 → 温度效果被干扰，还白烧 token。
