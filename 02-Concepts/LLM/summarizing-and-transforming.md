# 摘要与改写：Summarizing（压信息）vs Transforming（换外壳）

## 一句话解释

这是用 prompt 处理文本的两件事，判据就一把尺子——**信息量变不变**：

- **摘要 Summarizing = 压信息**：长 → 短，丢次要，信息**变少**（本质是「**有损压缩**」）。
- **改写 / 转换 Transforming = 换外壳**：信息**不变**，只换形式（语言 / 语气 / 格式 / 纠错）。

> 口诀：**摘要是减肥，改写是换衣服。**

## 摘要 vs 改写：用「信息量」尺子判（最重要的一张表）

| 任务 | 信息量 | 归类 |
|---|---|---|
| 会议记录 → 5 条待办 | 变少（丢讨论细节） | **摘要** |
| 5000 字长文 → 100 字简介 | 变少 | **摘要** |
| 中文通知 → 英文版 | 不变（只换语言） | **改写**（翻译） |
| 一段 JSON → HTML 表格 | 不变（只换形式） | **改写**（格式） |
| 抱怨语气 → 商务信函 | 不变（只换语气） | **改写**（语气） |

⚠️ 别凭「变长变短」判——改写也可能变长（口语→正式书面常更长），但信息没增没减，仍是改写。**只看信息量增减**。

## 控制摘要长度：三单位 + 软约束

prompt 里直接卡长度，三种「单位」都行：

| 单位 | prompt 写法 |
|---|---|
| 词数 | 「最多 30 个**词**」 |
| 句数 | 「最多 3 **句**话」 |
| 字符 | 「最多 30 个**字**」 |

⚠️ **这是「软约束」，不是分毫不差**——常会差几个。为什么？

1. 🔑 模型是**逐 token 边写边凑**的，**没有精确计数器在数数**，是「凭语感估着写」，天生做不到精确。
2. 你卡的是「字 / 词 / 句」，它内部是 **token**，两者**不一一对应**（中文按字、英文按词，算法还不同），更数不准。

➡️ **要严格卡死长度，光靠 prompt 不够，得在代码里再 `len()` 截一刀**（如 `text[:20]`）。
工程上更稳：prompt 留余量（让它「≤18 字」）+ 代码兜底截 + 按标点/句子边界截（别拦腰砍出半句话），或超了就让模型「重写一版更短的」。

> 🔑 **代码兜底 ≠ 约束模型**：`len()` 截断 / 数条数 的校验代码跑在模型**返回之后**，改变不了已经生成的输出，**不能反向「约束模型」**；它是**事后把关 + 补救**（不达标就截断 / 重试 / 只留前 N 条）。一句话：**prompt = 事前软请求（模型尽量做到），代码校验 = 事后硬保证（确定性兜底）**，两道关一软一硬。

管「范围/侧重」还有两招：**聚焦角度**（「聚焦价格相关」）和**提取**（见下）。

## 概括 summarize vs 提取 extract：求「全」还是求「专」

同一个需求，换个动词，效果差很多——因为两个动词的**默认任务目标不同**：

| | 默认目标 | 行为 | 适合 |
|---|---|---|---|
| **概括** summarize | **给全貌** | 即使加「聚焦物流」，本能仍要交代全局 → **会捎带**价格/质量 | 要**全貌总结** |
| **提取** extract | **做过滤** | 只留你点名那一类，其余**一律丢掉** → 干净专一 | 只要**某个具体维度**（运费/价格/某功能） |

> 🔑 「聚焦总结」压不住总结「要顾全局」的本能，所以仍会捎带；**要某个维度且要干净，用「提取」比「聚焦总结」可靠**。
> 比喻：概括=让同事「把整篇缩几句」（面面俱到）；提取=让同事「只挑关于物流的句子」。

## 判断摘要漏没漏重点：先定义重点 + 体检三件套

- **前提**：摘要是有损压缩，**一定会丢东西**；「漏」不可避免。真正的问题是「**丢的是不是你在乎的**」。
- 🔑 **没有绝对客观标准**——「重点」是**人为/主观定义**的，取决于**目的**（给老板看重点是结论、给法务看重点是风险，换用途重点就变）。所以**第一步永远是「先定义什么算重点」**。
- 查法（体检三件套）：
  1. **人工对照**：自己先凭原文列关键点 → 数摘要覆盖了几条，没覆盖的就是漏的。
  2. **提取反查**：对每个在乎的维度用「提取 XX 信息」各跑一次，和摘要对照（**提取 = 摘要的体检工具**）。
  3. **聚焦重摘**：怀疑某方面被漏，就专门「聚焦那方面」再摘一遍，和原摘要对比。

## 改写 / 转换的四类外壳

| 外壳 | 例子 prompt |
|---|---|
| **翻译** | 「把下面中文翻成英文」；还能「识别这段是什么语言」、一次翻多种语言 |
| **语气 / 风格** | 「把下面这段从口语改写成商务信函」 |
| **格式** | 「把这个 JSON 转成带表头的 HTML 表格」 |
| **纠错** | 「校对并修正下面文本的拼写/语法；若没错误就回复『无错误』」 |

- 多个转换能**串在一个 prompt** 里（翻译 + 改语气 + 输出 JSON）——这正是 `PR2-Gate 邮件处理器`的玩法。

## 实战：锁 JSON 的摘要 prompt（5 要素清单）

写「长文 → 3 要点 + 1 摘要、格式稳定」的核心 prompt，对着这 5 项**逐条打勾**：

1. **角色**：`你是信息摘要助手`（注意：用「中立提取/摘要」身份，别用「客服专员」之类会带回复语气的角色）。
2. **任务**：输出 3 条要点 + 1 段摘要。
3. **限长度**：`每条要点 ≤15 字，摘要 ≤50 字`。
4. **锁 JSON**：`格式为 {"points":[...],"summary":"..."}`。
5. **只输出 JSON**：`不要多余文字`（不加这句，模型可能回「好的，这是摘要：…」带一截废话）。

```text
你是信息摘要助手。读取下面 """ 之间的长文，输出 3 条要点和 1 段摘要，
只输出 JSON、不要多余文字，格式为 {"points":["...","...","..."],"summary":"..."}，
每条要点 ≤15 字，summary ≤50 字；若没有内容就输出 {"points":[],"summary":"无"}。
"""
<这里放长文>
"""
```

> 用 `"""..."""` 分隔符把「指令」和「待处理长文」隔开（好习惯，防长文里的句子被当成指令——prompt 注入防护）。

## 实证：PR2-02 机械版 vs 模型版（真跑）

同一段长文（远程办公，5 句），两版对比：

| | 机械版 `simple_summarize`（切句取前 3 句） | 模型版 `llm_summarize`（锁 JSON） |
|---|---|---|
| 要点 | 原文整句**照搬**、啰嗦 | 提炼到 10-12 字 |
| 覆盖 | 只取前 3 句、**漏掉后 2 段** | 3 条覆盖全文、摘要含「平衡点」 |
| 格式 | 不能 `json.loads` | `json.loads` 成功、正好 3 条、要点 ≤20、摘要 ≤60 |

🔑 取前 N 句的「机械摘要」= 原文照搬、会漏后文，**谈不上提炼**；模型版才是真摘要 + 可控格式。这次软约束（3 条 / ≤20 / ≤60）模型**全守住**，但仍要代码兜底——**守住是这次，不保证每次**。产物：`code/stage2/pr2_02_summarizer.py`。

## 真实场景应用

- **邮件处理器（PR2-Gate）**：邮件 → `分类 + 摘要 + 待办 JSON`。摘要环用「限长度」防复述全文 + 「聚焦」点名带上待办；**但关键的「待办/截止时间」别全押在摘要上**——它本就有独立的「待办 JSON」字段，用**提取**单独挖进该字段更不漏（双保险，呼应「提取比聚焦总结干净」）。
- **客服 agent 历史压缩**：每轮把超长历史先**摘要**成短摘要再喂模型，而非全量重发。省钱原因：历史是 [[multi-turn-stateless-memory|每轮都重发]]的，全量历史越聊越大、按全量收 `prompt_tokens`、滚雪球；摘要把每轮输入压在小上限内。这和「[[prompt-engineering-basics|few-shot 不能无脑堆几十个示例]]」是**同一个道理**——**往 context 里无脑塞 token，既费钱又稀释核心，长 ≠ 好，相关 + 精简才好**。

## 常见坑 / 错误理解 → 正确理解

- ❌ 凭「变长变短」判摘要/改写 → ✅ 只看**信息量增减**；改写可变长（口语→书面）但信息不变。
- ❌ 以为 prompt 写「最多 30 字」就严格 30 字 → ✅ 是**软约束**（模型无计数器、token≠字词）；严格卡死靠**代码 `len()` 截断**。
- ❌ 要某个维度却用「总结·聚焦 XX」、结果仍捎带别的 → ✅ 要干净就换动词「**提取** XX」（总结求全、提取求专）。
- ❌ 觉得「漏没漏重点」有客观标准、直接看摘要 → ✅ 「重点」是**主观、按目的定义**的，**先定义什么算重点**，再用提取反查/聚焦重摘核对。
- ❌ 写摘要 prompt 漏了「限长度」或「只输出 JSON」→ ✅ 对着 **5 要素清单逐条打勾**（角色/任务/限长度/锁 JSON/只输出 JSON）再交。
- ❌ 以为校验代码能「约束模型」→ ✅ 代码跑在模型返回**之后**，只能**事后把关 + 补救**（截断 / 重试），不能反向约束已生成的输出；prompt 是**事前软请求**，代码是**事后硬保证**。
- ❌ 锁 JSON 的 prompt 用 f-string 嵌正文 → ✅ JSON 示例的 `{}` 会被 f-string 当表达式（静默坏）/ `.format()` 报 KeyError；改**字符串拼接**（见 [[prompt-engineering-basics|Prompt 基础卡]]）。
