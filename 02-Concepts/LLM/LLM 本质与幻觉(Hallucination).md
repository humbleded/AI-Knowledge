# LLM 本质与幻觉

## 一句话解释

大语言模型（LLM）本质上像一个“超级文字接龙玩家”：根据当前可见内容，**预测下一个可能的 token（词块）**。这个训练目标本身不保证事实核查，因此幻觉不是一个靠修补单点 bug 就能彻底消除的问题；在 Agent 中，错误生成还可能被放大成真实行动。

## 地基：它只在干一件事

不管模型多大多复杂，每一步都只做一件事：

```text
看前文 -> 给所有候选词打概率 -> 挑一个"接上去最像样、最通顺"的词 -> 接回去，继续
```

抓住这条地基，第三章其它东西全是它的延伸：

- Transformer / 注意力机制 = 为了把"下一个词"猜得更准。
- Temperature / Top-p = 控制从候选词里"多稳 vs 多浪"地挑。
- 幻觉 = 这条地基的副作用(见下)。

## 为什么会幻觉（核心）

它优化的是「**下一个 token 在当前上下文中有多可能**」，但 **生成概率高 ≠ 外部事实一定真实**。

- 下一个 token 预测目标没有内置数据库查询或外部事实核验步骤。
- 模型可以学会表达“不确定”，但这种表达不等于概率已经校准，也不能保证每次都会主动停止猜测。
- 训练数据可能错误、矛盾、过时或带偏见；复杂推理中的早期错误也可能进入后续上下文并继续放大。
- 即使 Prompt 没有故意误导，模型对私有数据、实时状态或未见事实仍可能生成看似合理但未经核实的内容。

**直觉例子**：问它"我家那只猫叫什么名字？"——它不会说"我不知道"，多半会编一个像样的名字。因为"不知道"很少是"最像样的下一句"。

## 错误理解 -> 正确理解（易错点）

- ❌ 幻觉是模型"出错了 / bug" → ✅ 幻觉是"只求像、不求真"机制的**天生副作用**，无法靠"修 bug"根除，只能缓解。
- ❌ 只有数据有问题或被误导才会幻觉 → ✅ 数据干净、prompt 没问题时**照样**会幻觉。
- ❌ "上下文太长忘了前文"就是幻觉 → ✅ 那是**上下文窗口限制**(另一个问题)，会**加重**幻觉，但不是根因。详见 [[消息角色与指令优先级(Instruction Hierarchy)]]。

## 加重幻觉的诱因（不是根因，但会火上浇油）

- 训练数据里本身有错 / 互相矛盾(训练污染)。
- 误导性的 Prompt 把它带偏。
- 上下文太长，早期信息"顾头不顾尾"。

## 三种常见表现

| 类型 | 判断依据 | 例子 |
|---|---|---|
| 事实性幻觉（Factual Hallucination） | 与现实世界事实不符，或编造不存在的事实/实体 | 推荐不存在的航班 |
| 忠实性幻觉（Faithfulness Hallucination） | 没有忠实反映给定来源、检索文档或工具结果 | 工具返回“3 天到账”，却总结成“30 天” |
| 内在幻觉（Intrinsic Hallucination） | 与当前输入或上下文直接矛盾 | Observation 是 `timeout`，模型却说“支付成功” |

分类可能重叠：把 `timeout` 说成成功，既直接违背当前输入，也没有忠实反映工具结果。

## 为什么在 Agent 中更危险

普通聊天中的幻觉主要形成错误信息；Agent 还能把错误判断变成错误调用或真实副作用：

```text
错误生成 / 错误解释 Observation
  -> 错误 Tool Call、参数或状态判断
  -> 客户端若缺少硬校验
  -> 工具执行扣款、发货、删除等真实动作
  -> 错误状态进入下一轮继续放大
```

因此，模型说“成功”不能覆盖支付 API、订单系统或文件工具的结构化真实结果。外部系统状态的权威来源是相应工具/API，而不是模型生成的自然语言。

## 工程上怎么缓解

工程目标不是承诺“永不幻觉”，而是**降低发生概率，并阻止错误直接变成不可逆后果**：

| 防线 | 主要兜住什么 | 仍然不能保证什么 |
|---|---|---|
| System / Prompt | 引导模型引用证据、表达不确定性、遵守输出格式 | 属于软约束，模型仍可能不遵守 |
| RAG | 把可更新文档放进当前上下文 | 检索可能错，模型也可能曲解文档 |
| 外部工具 / API | 获取实时状态、精确计算或真实文件内容 | 模型仍可能错误解释返回值 |
| 客户端程序硬校验 | 强制 Schema、allowlist、权限、路径、业务状态机和重试规则 | 只能保护已经编码的边界 |
| 人工确认 | 在删除、扣款、发消息等高风险副作用前确认真实意图 | 不能替代事实来源与程序权限控制 |

还可以使用多步验证、外部评估、路由和 JIT 检索缩小上下文，但它们仍然是降低风险的方法，不是绝对正确保证。

### 超时必须保持“未知”

订单 API 返回 `timeout` 时，权威事实只是“本次查询超时”，并没有证明订单成功或失败。正确收口是：

1. 只在重试策略允许时有限重试。
2. 仍超时就停止查询并报告“目前无法确认订单状态”。
3. 禁止下游依据模型猜测执行扣款、发货等动作。
