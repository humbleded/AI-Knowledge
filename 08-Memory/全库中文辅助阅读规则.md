# 全库中文辅助阅读规则

## 核心原则

知识库可以对所有 Markdown 笔记提供中文辅助阅读，不限于 `raw/`。但不要把全库硬翻译成中文；应保留英文原文和技术锚点，在旁边补充中文理解。

最推荐的形式是：

> 原文保留，中文理解追加；术语保留英文，中文解释辅助；代码永不翻译。

## 翻译范围

可以翻译或补中文解释：

- 英文说明段落。
- 英文课程笔记。
- 英文论文/文章摘要。
- 英文报错信息的解释。
- 英文标题的中文别名。

不要翻译：

- 代码块。
- inline code。
- 命令行命令。
- 文件路径。
- API 名、类名、函数名、变量名。
- Git 分支、commit、package 名。
- URL 和 wikilink 目标。

## 专业词规则

专业词优先保留英文，并在首次出现或中文速读中加中文解释。

推荐格式：

- `backpropagation（反向传播）`
- `gradient descent（梯度下降）`
- `tokenization（分词 / token 化）`
- `embedding（嵌入向量）`
- `attention（注意力机制）`
- `transformer block（Transformer 模块）`
- `loss.backward()`：保留原样，只解释含义。

避免把英文术语完全替换成中文，因为后续阅读论文、源码、API 文档时，英文术语才是检索和理解的锚点。

## 推荐写法

对英文资料可以追加：

```markdown
## 中文速读

- ...

## 关键术语

- `term`（中文解释）：一句话解释。
```

对短英文段落可以使用 callout：

```markdown
> [!note] 中文理解
> 这里用中文解释上方英文段落，但保留关键英文术语。
```

## 自动化维护规则

每周知识库维护时，如果发现英文占比较高的笔记，应优先：

1. 跳过代码块、inline code、frontmatter、链接和命令。
2. 补 `## 中文速读` 或 `> [!note] 中文理解`。
3. 保留英文专业词，并用括号补中文。
4. 必要时把高频术语沉淀到 `02-Concepts`。
5. 更新 `log.md` 记录本次中文辅助阅读整理。
